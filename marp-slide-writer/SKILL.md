# Marp 章节增量写作

## Overview

- 先继承目标文档的结构、主题、分页和页面 class，再补写内容。
- 默认适配当前工作区常用的 `am_xmu` + `cover_e` 风格；如果目标文件已使用其他本地 theme，则保持原样。
- 仅在用户明确要求时，才重组整套幻灯片、切换主题或扩写为完整 deck。

## Workflow

1. 先检查目标 Marp 文档或相邻样例，确认这些事实：
   - frontmatter 中的 `theme`、`size`、`header`、`footer`、`headingDivider`
   - 是否使用 `cover_e`、`cols-2`、`cols-2-64`、`rows-2`、`tinytext` 等 class
   - 目标章节前后页面的粒度、语气、标题命名方式、图片和代码块写法
2. 明确本次任务是：
   - 补一页
   - 补一章（多页）
   - 重写现有一节
   - 将笔记/组会记录整理成局部汇报页
3. 只生成完成任务所需的最小改动：
   - 保持已有章节顺序和相对路径
   - 保持已有图片目录、资源引用方式、HTML 容器命名
   - 不主动改动无关页面
4. 写完后自查：
   - 每页只有一个核心结论
   - 标题和要点可以直接嵌入现有文档
   - 与前后页衔接自然，没有突然切换风格或术语

## Hard Constraints

- 优先编辑现有 Marp 文件；如果用户只给了笔记，也先按“要插入到已有汇报中的新页”来写。
- 默认语言使用中文，术语可中英混用，但必须与目标文档已有写法一致。
- 每页只表达一个核心信息，正文尽量压成 3–6 条要点，避免大段散文。
- 除非用户明确要求，不要重写封面、目录、参考文献页，不要把局部修改升级成完整 deck 重构。
- 保留目标文件已有的 theme；若没有可继承对象，再默认使用 `am_xmu`。
- 只有在目标文档明确使用其它本地 theme，或用户指定切换时，才改为 `/Users/jader/Work/研/themes` 下的其它主题。
- 保持原有相对路径，不要擅自改写图片目录、资源文件名、HTML 容器类名。
- 如果需要图片/代码/表格，优先沿用文档已出现过的页面组织方式。

## Common Page Patterns

### 本周进展 / 阶段进展

- 适合将组会记录、实验记录、TODO 列表整理成 1 页或 2 页。
- 标题优先使用“本周进展”“阶段进展”“某方向进展”。
- 每条要点写成“动作 + 结果/状态”，避免只有名词。
- 如果同页还要写计划，确保“已完成”和“下周计划”明显分组。

### 项目架构 / 设计页

- 先给一句总述，再展开模块、数据流、调度流程或代码结构。
- 如果已有架构图或目录树，优先使用双栏布局：一栏讲结构，一栏放图、树或流程。
- 代码树、模块树、组件职责适合配合 `cols-2` 或 `cols-2-64`。

### 技术实现 / 代码修改总结页

- 优先写真实修改点，不按Git时间线、提交链、分支名或 hash 展开，除非用户明确要求。
- 标题直接点对象和主题，例如“某模块设计”“某功能修改”，少用空泛标题。
- 正文更像组会里的开发记录，不像 PR 描述；先点模块、类、接口或状态流，再列改动。
- 优先使用“模块名 / 类名”单独起一小块，再列 2–4 条动作项；不要默认写成长句模板。
- 每条动作项只说一个改动，常见写法是“抽到……”“统一……”“增加……”“改成……”“补齐……”“支持……”。
- 少写“价值在于”“可以理解成”“这一轮不是……而是……”这类抽象且AI感过重的句；如果需要写影响，单独成一条。
- 不重复前文已经讲过的背景；默认只补新增信息。
- 对实现总结，优先落在代码里真实出现的结构变化，如类、状态、接口、测试覆盖，而不是泛泛总结“整体推进”。

### 问题分析 / 风险页

- 标题优先落在“问题”“挑战”“风险”“瓶颈”“局限”之一。
- 每条风险要包含现象、原因或影响，避免只列名词。
- 若文档风格允许，可在页尾补一行“应对策略 / 后续动作”。

### 实验结果 / 调研总结

- 只保留支撑当前结论的结果，不堆完整实验日志。
- 用一句话先点明结论，再给 2–4 条证据。
- 如果配图，图注要能解释“为什么这张图值得放在这里”。

### 下周计划 / 后续安排

- 用明确动作开头，如“完成”“实现”“验证”“阅读”“对比”“定位”。
- 计划项数量保持克制，优先列最关键的 2–4 件事。
- 与前一页进展呼应，避免出现与当前问题无关的新方向。

## Theme And Style Rules

- 目标文件已有 frontmatter 时，优先继承它的 `theme`、`header`、`footer`、`headingDivider`、`math` 等设置。
- 新建局部页面但没有明确目标文件时，默认参照工作区常见写法：
  - `marp: true`
  - `theme: am_xmu`
  - `size: 16:9`
  - 视现有文档决定是否开启 `math`
- 仅在新增封面页时才考虑 `cover_e`；普通补页不主动插入封面或目录页。
- 如果相邻页面使用 HTML `<div class="ldiv">`、`<div class="rdiv">` 等容器，新增页保持同一套容器命名。
- 如果目标文档本身偏“组会记录”风格，则优先贴近这种写法：对象明确、动作直接、少解释性套话。

## Boundaries

- 本 skill 不负责：
  - 默认生成完整汇报 deck
  - 自动抓取论文/arXiv/PDF 内容
  - 收集 `.assets`、转换图片、导出 PPT
- 如果任务已经变成“从资料自动整理完整汇报 deck”，改用 `marp-report-deck` 更合适。
